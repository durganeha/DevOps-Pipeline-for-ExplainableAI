# Building a DevOps Pipeline for Explainable AI (XAI)
### An Automated System for Validating Model Fairness — Loan Approval Domain

An automated CI/CD pipeline that blocks a machine learning model from being
deployed unless it passes a fairness check, explains its decisions using
SHAP, and continuously monitors for bias drift after deployment.

---

## Why this exists

Machine learning models used for high-stakes decisions (loan approval,
hiring, lending) can be biased against protected groups — and that bias can
creep in *after* deployment even if the model was fair when it first
launched, as the real-world population shifts. Most deployment pipelines
only test "does the code work," not "is the model still fair." This project
adds that missing check directly into the DevOps pipeline.

## What it does

1. **Fairness Gate (CI/CD)** — every push automatically trains/evaluates the
   model and computes its Disparate Impact (DI) score. If DI falls below the
   0.8 legal threshold (the "80% rule" used in US anti-discrimination law),
   the CI job fails and blocks the merge.
2. **Explainability (XAI)** — SHAP explains *why* the model approved or
   rejected a given applicant, and which features matter most overall.
3. **Drift Monitoring** — a CUSUM (Cumulative Sum) statistical detector
   simulates incoming applications over time and flags when the *deployed*
   model's fairness degrades due to real-world data drift — without the
   model itself changing.

## Architecture

```
                    ┌─────────────────────┐
   Push / PR   ───▶ │  GitHub Actions CI    │
                    │  1. Run unit tests    │
                    │  2. Train + evaluate  │
                    │  3. Fairness Gate     │──▶ ❌ blocks merge if DI < 0.8
                    │  4. CUSUM drift check │──▶ ✅ uploads report artifacts
                    └─────────────────────┘

                              ▼
                    ┌─────────────────────┐

                    │  🗂️ Pipeline History   │
                    └─────────────────────┘

│   ├── fairness_metrics.py               # DI, equal opportunity, statistical parity

├── models/baseline_model.pkl             # trained model artifact

├── Dockerfile
└── requirements.txt
```

## Getting the real dataset

This project ships with a schema-matching **synthetic fallback dataset**, so
everything runs out of the box. To use real data:

1. Get the **Loan Prediction Problem Dataset** (Analytics Vidhya / Kaggle,
   uploader `ninzaami`)
2. Download `train.csv`, rename it to `loan_data.csv`
3. Place it at `data/raw/loan_data.csv`
4. Nothing else changes — `src/data_loader.py` automatically prefers the real
   file over the synthetic one whenever it's present.

## Running locally

```bash
pip install -r requirements.txt

# Train the model (uses real data if present, else synthetic fallback)
python src/train_model.py

# Run the fairness gate manually
python scripts/fairness_gate.py --threshold 0.8

# Run the drift simulation
python scripts/cusum_drift_detection.py

# Run the test suite
pytest tests/ -v

# Launch the UI
streamlit run app.py
```

## Running with Docker

```bash
docker build -t xai-fairness-pipeline .
docker run -p 8501:8501 xai-fairness-pipeline
```
Then open http://localhost:8501.

## Fairness metric used

**Disparate Impact (DI)** = P(approved | unprivileged group) / P(approved |
privileged group). A DI of 1.0 is perfectly fair; the widely used legal
threshold for concern is **DI < 0.8** (the "80% rule"). This project also
implements Equal Opportunity Difference and Statistical Parity Difference
(`src/fairness_metrics.py`) for a fuller picture.
